# Area Designs (full) — Impersonation, Comments, Process
Detailed field/contract specs behind the master plan. Source of truth for implementation.

## Password auth + traceable user impersonation ("act as") across the Actor model, executor, Tree Event, and Change Request audit trail — with an SSO/password seam that keeps the public repo OSS-clean.

**Summary:** The auth seam already exists and is mature: arbor/auth/ ships an AuthProvider Protocol + LocalAuthProvider (Frappe-native username/password) + OIDCAuthProvider, selected by site config arbor.auth.provider_class and resolved by get_auth_provider(); the AfterShip SSO overlay conforms structurally in the private repo with zero core coupling (guarded by tests/auth/test_auth_seam.py). So AREA 1(a) is largely DELIVERED for the backend — the remaining public work is a login SCREEN + a whoami-driven auth gate in the frontend shell, both provider-agnostic.

The core of this design is (b) TRACEABLE impersonation, built as a request-scoped OVERLAY that is deliberately DECOUPLED from authentication. The insight: never call frappe.set_user() to "become" someone — that would make frappe.session.user itself lie and destroy the real-user trace at the framework boundary. Instead the REAL Frappe session (established by whatever provider authenticated: password or SSO) stays authoritative, and impersonation is a separate admin-gated, server-persisted overlay that arbor/arbor/api.py::_actor() reads to build Actor(user=<impersonated>, real_user=<real session user>, impersonated_as=<impersonated>, actor_type, is_admin). ACL then runs against the EFFECTIVE user (so admins genuinely see each role's view), while both identities travel through the ONE executor into every Tree Event and Change Request. This adds two fields to the frozen core Actor dataclass, two columns to Tree Event, two columns + a table field to Change Request, and stamps them in executor._run_and_emit / _suggest / _suggest_batch and change_request.create_* — so an impersonated action is NEVER confusable with a real login: real_user is populated and != actor, and actor_type gains an "impersonation" discriminator on the event.

(c) The seam: impersonation sits ABOVE the AuthProvider (auth resolves the real user; impersonation overlays an effective user). It references no provider, so password (public LocalAuthProvider) and SSO (private AfterShipSSOProvider) both feed it identically. The frontend impersonation UI is provider-agnostic and lives in the public shell; the private overlay keeps ONLY its SSO wrapper. New capabilities beginImpersonation / endImpersonation are registered in the registry + hooks override_whitelisted_methods and admin-gated in the executor exactly like internalReset / assignRole.

### aclImplications

Two-layer authority. Layer 1 (authenticate): the AuthProvider resolves the REAL Frappe session user (password via LocalAuthProvider, or SSO via the private provider). Layer 2 (impersonate overlay): applied in _actor(). The ordering is load-bearing: _actor() must (1) read frappe.session.user = real_user, (2) compute real_is_admin from the REAL user's roles, (3) look up the active Arbor Impersonation Session for real_user, (4) if present AND real_is_admin, build Actor(user=impersonated, real_user=real, impersonated_as=impersonated, is_admin=<recompute from IMPERSONATED user so the admin genuinely experiences that user's affordances>). If the overlay exists but the real user is no longer admin (grant revoked mid-session), the overlay is IGNORED and force-ended (fail-safe: you cannot retain someone else's identity by losing admin). beginImpersonation/endImpersonation authority is gated on real_is_admin — computed before the overlay is applied — so an impersonated non-admin identity can NEVER start a nested impersonation or escalate. The existing axis ACL (structural/branch + per-column) and role-admin gates are UNCHANGED: they run against Actor.user (the effective identity), which is exactly the point — the admin sees precisely what that owner sees, edits route/suggest exactly as they would for that owner. Read-ACL (visible_columns) likewise runs on the effective user. internalReset stays admin/system only and is additionally blocked while impersonating (defense: an impersonated identity has is_admin only if the impersonated user is themselves admin, so this is naturally safe, but we assert real_user==user for internalReset to avoid an admin nuking data under someone else's name).

### llmSurface

The server Re-Act agent MUST NOT be able to impersonate: beginImpersonation/endImpersonation are registered with is_exposed_to_llm=False, so they never enter the agent's tool catalog (the agent builds tools from registry entries flagged for LLM). Additionally the executor admin-gate (_ADMIN_IMPERSONATION_CAPS) requires actor.is_admin, and the agent runs as its own non-admin ActorType.AGENT user, so even a coerced call is rejected with AuthorizationError. Conversely, the agent is unaffected BY impersonation: when an admin impersonates and then invokes the agent, the agent still runs as its own agent user (actor_type=AGENT), not as the impersonated identity — impersonation is a human-surface concept resolved in _actor() and does not propagate into the agent's own Actor construction. This keeps the "agent uses its OWN user" invariant intact and prevents an impersonation window from silently re-attributing agent actions.

### publicVsPrivate

PUBLIC (arbor-treetable, OSS-clean): everything in this design except the SSO wrapper. The Actor fields, Tree Event/CR columns, Arbor Impersonation Session doctype, beginImpersonation/endImpersonation capabilities, the executor gates, _actor() overlay logic, whoami extension, LoginScreen, ImpersonationBar, useWhoami, and all api.ts additions are provider-agnostic and contain zero AfterShip/automizely/SDK strings — they pass tests/auth/test_auth_seam.py::test_no_sso_overlay_sdk_import_in_core unchanged. Username/password login = Frappe-native /api/method/login via LocalAuthProvider (the default when no provider_class is configured). PRIVATE (arbor-sso-aftership): unchanged — it keeps ONLY AfterShipSSOProvider (server JWT verify) and authProviderEmployee.tsx (frontend SDK bridge). Impersonation needs NO private-side code: because it overlays the resolved real user and the SSO provider already lands the real employee as frappe.session.user, _actor()'s overlay works identically under SSO. The only private touch, if desired, is that the SSO build may set admin_emails (already supported) to decide who may impersonate — reusing the existing is_admin signal, not a new hook.

### risks
- Silent set_user pitfall: if a future contributor 'simplifies' impersonation to frappe.set_user(impersonated), the real-user trace is destroyed and the audit becomes indistinguishable from a real login. The design explicitly forbids this; a bench test asserting frappe.session.user stays == real_user during impersonation locks it in.
- Actor is a frozen dataclass used pervasively; adding fields with defaults is backward-compatible, but every construction site that positionally builds Actor must be checked (grep shows Actor(...) is built by keyword in api.py and by cr re-resolution with a bare user — safe because new fields default to None).
- Admin-revocation race: an impersonation session persisted for a user who loses admin must not grant lingering foreign identity. Mitigated by recomputing real_is_admin every request and force-ending the overlay if lost — but this must be covered by a bench test, else it silently regresses.
- Existing Tree Event append-only guard + track_changes: adding columns is fine, but the migration must not trip the on_update/on_trash append-only controller guard; new columns are set only at insert time in FrappeEventSink.emit, so no update path is introduced.
- OSS-clean regression: LoginScreen/ImpersonationBar must contain no vendor strings; test_auth_seam already scans frontend/ for SDK imports but NOT for brand strings — recommend widening that guard is out of scope here, so reviewers must keep the new files generic.
- Parity/registry drift: two new capabilities require updating the parity manifest and registry test in the SAME change, or the parity bench fails (standing constraint).

### openQuestions
- Who may impersonate whom — is any System Manager allowed to 'act as' ANY user, or should there be a narrower allowlist / a rule barring impersonating another admin? Current design = any is_admin may impersonate any enabled non-... (recommend: forbid impersonating another admin to prevent privilege-laundering; needs product confirmation).
- Should impersonation be time-boxed (auto-expire after N minutes) via the started_at field + a scheduler tick, or persist until explicitly ended? Design leaves ended_at nullable and supports both; default is explicit-end.
- Do we want a dedicated IMPERSONATION discriminator on Tree Event.actor_type (a 4th enum value) instead of relying on real_user!=user? Chosen NO to keep the closed actor_type set stable and agent/system semantics untouched, but if reporting wants a first-class filter this is a small additive change.
- Should endImpersonation also emit a lightweight audit Tree Event (e.g. a new session-scoped event) or is the Arbor Impersonation Session row (with started_at/ended_at) sufficient as the audit record? Current design keeps begin/end OFF the 11-event closed set and relies on the session doctype — confirm that satisfies auditors.
- For the public password login, do we want to also expose Frappe's built-in /login page (via LocalAuthProvider.get_login_url) instead of a custom LoginScreen, to avoid reimplementing password reset/2FA flows? Design offers the custom screen for a unified look but the provider already points at /login — product/UX call.

## AREA 2 — per-cell comments via a right-side drawer

**Summary:** Add threaded per-cell comments as a NON-capability, cell-keyed feature that mirrors the shipped "Arbor Cell Draft" precedent (api.py L1233-1377, hooks L115-121): a new `Arbor Cell Comment` DocType keyed by (sheet, node, column) with self-referential threading, author, timestamp, body, resolve/unresolve and @mentions; a set of thin `@frappe.whitelist()` read/write shims in arbor/arbor/api.py (add/list/resolve/reopen/delete) registered in hooks.override_whitelisted_methods; and a right-side `CommentDrawer` React component that opens for the selected cell.

Comments deliberately do NOT enter the registry (they are UI collaboration metadata, not a governed tree mutation) and do NOT emit Tree Events (the closed 11-EventType set stays intact) — exactly the stance the drafts/activity/roles-read shims already take. Governance is layered on top: read/post authority reuses arbor.core.acl.can_read_column (you can discuss any cell you can read); resolve/reopen authority reuses resolve_column_approvers (column owner + editors) so only the people who own the data can mark a discussion settled; delete is author-only OR column-approver. This keeps the ONE ACL resolver as the single source of truth — no new resolver.

Notifications reuse the EXISTING Notification/Acknowledgement DocTypes but via a DIRECT create path (repo.create_notification / frappe.get_doc) at comment-post time, NOT the Tree-Event→subscription dispatcher (a comment is not a Tree Event). We make Notification.tree_event nullable and add a `source` discriminator + a `comment` Link so the same inbox surfaces comment notifications; recipients = the cell's column_owner + editors + every @mentioned user who can still read the column (never the author). The snapshot gains a per-cell `comments` summary map ({(node,column): {open, resolved, unread}}) computed in api._acl_hints-adjacent code (mirroring _pending_cell_marks) so the grid shows an unread/comment-count affordance with zero extra round-trips and never leaks a column the viewer can't read.

Frontend: a cell affordance (a small comment glyph, sibling to the existing pending/draft markers in cells/Cell.tsx) sets a `selectedCell` in App.tsx and opens a right-side drawer. The drawer coexists with the floating agent widget (bottom-right FAB, unchanged) and the Live/Proposed overlay by being a LEFT-anchored-content, RIGHT-edge panel that pushes nothing (fixed, high z-index below the agent popup) and is read-only in Proposed preview mode (like cells). Optional, recommended: expose comments to the server Re-Act agent as READ-ONLY via a dedicated non-registry method so the agent can cite discussion without gaining a governed write tool.

### endpointsOrCapabilities
- `arbor.add_cell_comment(sheet, node, column, body, parent_comment=None)` → whitelisted shim in arbor/arbor/api.py. Actor = _actor(). AUTHZ: can_read_column(repo, sheet, col, actor) else frappe.PermissionError (403). Validates non-empty body (else 400 ValidationError); validates cell exists (unknown node/column → 404). Derives thread_root (parent's root, or self on a new root). Parses @mentions from body, resolves to User names, drops any who cannot read the column. Inserts Arbor Cell Comment, then DIRECTLY creates Notification rows (source=comment) for column_owner+editors+mentions minus author. Returns {name, thread_root, mentions}.
- `arbor.list_cell_comments(sheet, node, column)` → GET-style read shim (like list_cell_drafts). AUTHZ: can_read_column else 403. Returns ordered thread list: [{name, thread_root, parent_comment, author, body, mentions, resolved, resolved_by, resolved_at, timestamp, can_resolve, can_delete}] where can_resolve = actor in resolve_column_approvers, can_delete = actor==author OR actor in resolve_column_approvers. Ordered by creation asc, grouped by thread_root client-side.
- `arbor.resolve_cell_comment(comment, resolved=True)` → shim. AUTHZ: actor in resolve_column_approvers(repo, sheet, column) (column owner+editors) else 403. - `arbor.delete_cell_comment(comment)` → shim. AUTHZ: actor==author OR actor in resolve_column_approvers else 403. Soft-safe: if the comment is a thread_root with replies, blank the body + mark `[deleted]` rather than orphaning replies; a leaf reply is hard-deleted. Returns {ok:True}.
- `arbor.list_cell_comment_notifications(sheet)` (OPTIONAL / or fold into existing list_notifications): extend arbor.list_notifications to also surface source=comment rows for the sheet (join Notification→Arbor Cell Comment→its sheet), returning event_type='COMMENT_ADDED' (display-only string, NOT an EventType) so the existing NotificationItem renders them. Preferred: fold into list_notifications so there is ONE inbox.

## AREA 3 — Column-order PROCESS management + per-transition SLA + per-process Kanban/flow dashboard + per-user cross-sheet Notification Inbox page. All OSS-public (public repo victor-develop/arbor-treetable); zero SSO/AfterShip coupling.

**Summary:** A "process" is a per-sheet ordered list of column stages (A -> B -> C). When enabled, each ROW (a node, or a chosen node depth) gets a per-row Process Run tracking which stage is active and the timestamp each stage was entered/filled. Stage advancement is detected NOT by inventing new Tree Events (the 11-type set is CLOSED) but by a new dispatch-lane consumer that reacts to the SAME Tree Event stream the notification/webhook dispatchers already consume: NODE_CREATED starts a run at stage A and notifies A's column owner; a NODE_VALUE_UPDATED whose payload.column == the run's current stage column advances the run to the next stage and notifies that owner; the terminal fill completes the run. Per-transition SLA is a duration on each stage; a stage's due_at = entered_at + sla; a scheduler sweep marks runs breached and (optionally) notifies. The design reuses the existing notification machinery end-to-end: process notifications are created via the SAME repo.create_notification direct-recipient path the role_app flow uses (in-app channel, idempotent per (tree_event, recipient)), so the per-user Inbox and per-sheet notification list light up for free. Process definition + enable/disable + advance are governed CAPABILITIES routed through the ONE executor (defineProcess/enableProcess/disableProcess are META/structural-owner-gated; the runtime advance is a dispatch-lane side effect, not a user capability). The Kanban dashboard is a read-only whitelisted aggregate over Process Run rows; the Inbox is a new whitelisted read that reuses list_notifications logic but ACROSS sheets. Pure stage-machine logic lives in a new framework-free arbor/core/process.py module (unit-testable with the in-memory doubles); the Frappe binding lives in the dispatch lane + a Process Run doctype.

### risks
- Stage-advance detection off NODE_VALUE_UPDATED must NOT double-count: the same cell edited twice on the current stage should advance once; the notified_owner/filled_at guard on Process Run Stage plus the dispatcher's per-(tree_event,run) idempotency (mirroring notification_exists) must be airtight, or owners get spammed.
- Defining the 'row' unit is ambiguous — a tree is not a flat table. row_scope default 'root-children' is a pragmatic choice but may not match every sheet; 'all-nodes' could create runs for structural/group nodes that were never meant to be process rows. Needs a clear default + owner override, and enableProcess backfill must respect it.
- Ordering coupling: stage order is by column, but a NODE_VALUE_UPDATED can fill columns out of order (owner B fills before A). Decide policy: strict (out-of-order fill does not advance, only current stage counts) vs lenient (fill advances to the highest contiguous filled stage). The design assumes STRICT for determinism; product may want lenient.
- SLA sweep frequency vs scheduler load: a per-minute sweep over many active runs needs an index on (status, current due_at); the query must be bounded (only active runs with a non-null due_at in the past).
- Reusing COLUMN_CONFIG_UPDATED for process define/enable means the activity feed + subscriptions see process changes as column-config events — acceptable (keeps the closed event set) but the op discriminator must be honored by any consumer that cares.
- Emitting NO Tree Event for stage transitions means process advancement is invisible to webhooks/activity feed; if external systems need to react to 'stage entered', that is a follow-up (would require either a new event type — breaking the closed set — or a webhook off the notification, which is out of scope).
- inbox() cross-sheet query could be heavy for prolific users; needs pagination (keyset like list_activity) and an index on Notification.recipient+creation.

### openQuestions
- Out-of-order fills: strict (only current stage advances) or lenient (advance to highest contiguous filled)? Design assumes strict; confirm product intent.
- What is a 'row' by default — direct children of root, all leaf nodes, or nodes at a configured depth? Design defaults to root-children with an override; confirm.
- Should an SLA breach ESCALATE (notify the structural owner / re-notify) or only mark red on the dashboard? Design marks + optionally notifies the stage owner once; escalation chain is a possible extension.
- Multiple concurrent processes per sheet, or exactly one enabled at a time? Design enforces exactly one enabled process per sheet for clarity; confirm whether parallel processes are needed.
- Does completing a run require the LAST stage's column to be filled, or a separate explicit 'done'? Design uses last-stage fill = completed.
- Should stage transitions be replayable/derivable purely from the Tree Event log (so a run can be reconstructed after the fact), or is the Process Run the sole source of truth? Design treats Process Run as source of truth but the log is sufficient to rebuild — confirm if rebuild-on-enable of historical timings is required.
- SLA units — expose seconds (design) or business-days/hours honoring a calendar? Business-hours SLA would need a calendar model (larger scope).
